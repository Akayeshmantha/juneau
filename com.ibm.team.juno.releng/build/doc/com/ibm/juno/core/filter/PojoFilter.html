    public abstract class PojoFilter<T,F>
extends Filter
    Used to convert non-serializable objects to a serializable form.
    Description

    PojoFilters are used to extend the functionality of the serializers and parsers to be able to handle POJOs that aren't automatically handled by the serializers or parsers. For example, JSON does not have a standard representation for rendering dates. By defining a special Date filter and associating it with a serializer and parser, you can convert a Date object to a String during serialization, and convert that String object back into a Date object during parsing.

    Object filters MUST declare a public no-arg constructor so that the bean context can instantiate them.

    PojoFilters are associated with instances of BeanContexts by passing the filter class to the BeanContextFactory.addFilters(Class...) method.
    When associated with a bean context, fields of the specified type will automatically be converted when the BeanMap.get(Object) or BeanMap.put(String, Object) methods are called.

    PojoFilters have two parameters:

    1. <F> - The filtered representation of an object.
    2. <T> - The normal representation of an object.

    Serializers use object filters to convert objects of type T into objects of type F, and on calls to BeanMap.get(Object).
    Parsers use object filters to convert objects of type F into objects of type T, and on calls to BeanMap.put(String,Object).
    Filtered Class Type <F>

    The filtered object representation of an object must be an object type that the serializers can natively convert to JSON (or language-specific equivalent). The list of valid filtered types are as follows...

    • String
    • Number
    • Boolean
    • Collection containing anything on this list.
    • Map containing anything on this list.
    • A java bean with properties of anything on this list.
    • An array of anything on this list.
    Normal Class Type <T>

    The normal object representation of an object.

    One-way vs. Two-way Serialization

    Note that while there is a unified interface for handling filtering during both serialization and parsing, in many cases only one of the filter(Object) or unfilter(Object, ClassMeta) methods will be defined because the filter is one-way. For example, a filter may be defined to convert an Iterator to a ObjectList, but it's not possible to unfilter an Iterator. In that case, the generalize(Object} method would be implemented, but the narrow(ObjectMap) object would not, and the filter would be associated on the serializer, but not the parser. Also, you may choose to serialize objects like Dates to readable Strings, in which case it's not possible to reparse it back into a Date, since there is no way for the Parser to know it's a Date from just the JSON or XML text.

    Additional information
    See com.ibm.juno.core.filter for more information.
